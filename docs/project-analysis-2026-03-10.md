# Анализ проекта APIfy2 (по PRD, плану и текущей реализации)

Дата: 2026-03-10

## 1) Краткий вывод

Проект находится в рабочем состоянии как **API-тестер с AI-ассистентом**, но между документацией и фактической реализацией есть расхождения:

- `docs/development-plan.md` помечает фазы 1–10 как завершенные (с отдельными “future” пунктами).
- `docs/current-status.md` и `docs/progress-update.md` частично устарели и описывают более раннее состояние.
- В коде присутствуют ключевые возможности (auth, ручное тестирование, история, избранное, OpenAPI import, AI-провайдеры, CORS proxy), но вкладка аналитики на дашборде выглядит как заглушка.

Итог: ядро продукта реализовано, но требуется синхронизация документации и точечная стабилизация (схема БД vs runtime-таблицы, закрытие технического долга, unit/e2e-проверки).

## 2) Соответствие PRD и плана

### Что совпадает с PRD/планом

- Есть пользовательская аутентификация через Supabase.
- Есть ручное тестирование API (метод, URL, заголовки, body, auth).
- Есть импорт OpenAPI/Swagger и извлечение эндпоинтов.
- Есть история тестов и избранное.
- Есть AI-интеграция и выбор AI-провайдера.

### Что требует выравнивания

- По плану фаза аналитики отмечена как завершенная, но в текущем UI вкладка “Аналитика” содержит placeholder-текст.
- В документах фигурируют имена файлов/компонентов, которых уже нет или они переименованы (например, эволюция от `ApiTestForm` к `UnifiedApiTester`).
- Часть проектной документации описывает этапы как “будущие”, хотя в коде часть функций уже присутствует.

## 3) Архитектура и структура (фактическая)

### Frontend

- Next.js App Router (`app/`), основные пользовательские сценарии в `/dashboard`.
- Главный рабочий контур: `Dashboard -> Header + MainWorkspace`.
- Внутри `MainWorkspace` вкладки:
  - testing (`UnifiedApiTester`)
  - favorites (`FavoritesTab`)
  - import (`OpenApiImport`)
  - analytics (пока заглушка)
  - history (`HistoryTab`)

### Backend/API

- Route Handlers в `app/api/*`.
- AI-запросы маршрутизируются через `/api/ai/analyze`.
- Есть отдельный прокси-роут `/api/proxy` для обхода CORS в клиентских сценариях.

### Data layer

- Supabase клиент разделен на browser/server/middleware helpers.
- SQL-схема в `supabase/schema.sql` содержит базовые таблицы (`users`, `api_tests`, `test_history`, `api_documentation`) и RLS-политики.

## 4) Наблюдения по качеству и рискам

1. **Согласованность БД и кода**
   - В runtime-логике история тестов работает с таблицей `api_test_history`, тогда как в `schema.sql` определена `test_history`.
   - В runtime используется таблица `favorites`, которой нет в текущем `schema.sql`.
   - Риск: проблемы при “чистой” установке проекта по SQL из репозитория.

2. **Документация расходится с реализацией**
   - В проекте одновременно есть документы с разной стадией готовности (завершено/не завершено).
   - Риск: команда и новые контрибьюторы получают противоречивую картину.

3. **Аналитика как зона технического долга**
   - Вкладка аналитики в основном workspace выглядит незавершенной для уровня “фаза завершена”.
   - Риск: mismatch ожиданий продукта и фактического UX.

4. **Тестовое покрытие и валидация**
   - Нет отдельного скрипта тестирования (unit/integration/e2e) в `package.json`.
   - Риск: регрессии в критичных потоках (AI parsing, OpenAPI parsing, auth, history).

## 5) Рекомендации (приоритет)

### P0 (сразу)

1. Синхронизировать `supabase/schema.sql` с фактическими таблицами, используемыми в коде (`api_test_history`, `favorites`) **или** привести код к текущей схеме.
2. Обновить “источник правды” по статусу проекта:
   - выбрать 1 документ со статусом (например, `docs/current-status.md`),
   - остальные привести к ссылке на него.
3. Зафиксировать статус аналитики: либо реализовать рабочий минимум, либо явно пометить как in progress во всех документах.

### P1

4. Добавить минимальный набор тестов:
   - unit: парсинг/нормализация AI JSON,
   - integration: OpenAPI parser,
   - smoke e2e: login -> manual test -> save history.
5. Добавить миграции (SQL по шагам), чтобы схема не зависела от “ручных правок”.

### P2

6. Сократить дублирующие и устаревшие markdown-файлы, оставить компактный docs-индекс.
7. Добавить архитектурную схему “как данные проходят от UI до Supabase/API routes”.

## 6) Оценка зрелости

- **Продуктовая зрелость**: средняя/выше средней для MVP+.
- **Техническая зрелость**: средняя (видны функциональные возможности, но есть несогласованность docs/schema/testing).
- **Готовность к масштабированию**: условная; после закрытия P0/P1 задач — хорошая база для следующего релиза.
